  ★ Insight ─────────────────────────────────────
  One deliberate departure from the sibling config: sanctissimissa-app serves
  its .db with Cache-Control: public, immutable, 1y — but /missal.db keeps the
  same URL across releases (it's not content-hashed like Vite's assets), so an
  immutable cache would pin returning visitors to a stale corpus for a year
  after an ingest. I gave it public, no-cache + ETag instead: browsers still
  cache the 97 MB file, but revalidate with a cheap conditional request and only
  re-download when the corpus actually changes (nginx's ETag is derived from
  mtime+size).

  What it would take to architect in — and the two decisions that shape
  everything:

  1. Where the intelligence lives. This app is deliberately a zero-backend
  static SPA — the whole corpus is client-side in sql.js. A chatbot has three
  options, in increasing order of breaking that property:
  - Corpus-native, no LLM — the retrieval machinery already exists and is
  genuinely good: CorpusDb exposes FTS5 search, vector neighbours
  (groupedSimilarToText), and the concept taxonomy (conceptsForText,
  groupedConcordance). A chat UI over those is answer-retrieval, not generation,
  and stays fully offline/static — deploys stay rsync-and-symlink.
  - LLM via an API endpoint — an nginx /api/ proxy to a backend (the
  sanctissimissa pattern, port 9837 style, or straight to an inference box on
  the LAN). First server-side component; the collinear rule would need a stated
  exception since Tauri/Android builds would need the same endpoint or a local
  model.
  - On-device model — heavy for web (WebLLM-class downloads), plausible for
  Tauri native.

  2. What "DOM access" means here. For a same-app widget, it isn't
  third-party-embed DOM scraping — the bot lives inside the React tree and gets
  state access instead, which is strictly better: current route, resolved
  liturgical day, active station/section id, current text selection. That
  context object + corpus retrieval is what makes answers like "what is this
  prayer I'm looking at" work. Corpus access is trivially available since
  CorpusDb is already a singleton in the page.

  Architecturally it's one new UI entity (ChatDock — floating bottom-right
  button + panel, rendered in App.tsx above the router so it persists across
  views), one context-provider entity (view/selection state), and one
  answer-engine entity behind an interface so option A can be swapped for option
  B without touching the UI — that last split matters because it means you can
  ship the corpus-native version now and upgrade to an LLM later with no UI
  rework.

  DOCS/ARCHITECTURE.md and CHECKLIST stanzas before any code. Want me to draft
  that contract addition, and if so: corpus-native first, or straight to the
  LLM-backed design?

● That reframes it completely — a companion, not a concordance. The retrieval
  machinery stays out of the foreground; instead it's a conversational partner
  that knows where you are in the liturgy and what you've been writing, for
  reflection and idea-bouncing. That's an LLM-backed design by necessity —
  generation, not lookup.

  The good news is the app already has the two context anchors this needs:

  - Liturgical situatedness — the resolved LiturgicalDay (today's feast, season,
  rank, commemorations) and the active station/reading are already first-class
  state. The companion's system prompt gets assembled from real corpus rows:
  "the user is reading the Collect for Dominica VII Post Pentecosten…" —
  grounded discussion, not generic chat.
  - Journaling substrate — src/core/annotations/store.ts already exists
  (localStorage v1, schema deliberately mirrors a future sync table). Journal
  entries are a natural sibling record type there, and the companion can be

  Journaling: substrate yes, journal no

  What's implemented is the annotations sidecar (src/core/annotations/store.ts)

  Edge LLM landscape, mid-2026

  Gemma 4 E2B on LiteRT-LM is the clear fit, and it's better than what I'd have
  told you from training data:

  - Gemma 4 E2B — 2.3B parameters purpose-built for edge, runs in under 1.5 GB 
  RAM at 2/4-bit quantization, and benchmarks ahead of Gemma 3 27B despite being
  12× smaller. That's genuinely enough model for reflective discussion, not
  just autocomplete.
  - LiteRT-LM (Google's production edge runtime) now supports Multi-Token 
  Prediction drafters with Gemma 4 — up to 2.2–3× inference speedup — and runs
  on Android, iOS, Web, Desktop, and IoT from one framework.
  - The web story is real now: WebGPU-backed LiteRT-LM/MediaPipe LLM Inference
  runs client-side in the browser at up to ~76 tok/s decode on a MacBook-class
  machine — serverless, private, zero per-call cost.

  The strategic fit for this codebase is striking: one runtime family covers
  your entire Tauri target matrix — web/PWA (WebGPU), Android (native
  LiteRT-LM), desktop (LiteRT-LM desktop). That's the collinear rule extended to
  inference: one CompanionEngine everywhere, platforms differing only in how
  the runtime is loaded — exactly how loadCorpus.ts already treats the database.
  (Qwen's small models are competitive on raw quality but don't have that
  unified web+mobile runtime; you'd be stitching llama.cpp/MLC per platform.)

● Fair challenge — I over-applied the rule. Let me separate what the rule
  actually says from what I imported into it.

  What the no-placeholder rule actually forbids (Decision 6 in the
  architecture): UI surfaces must never render fabricated content presented as 
  corpus data — no lorem-ipsum propers, no invented antiphons, and anything
  gap-filled gets flagged meta.filled so supplied text is never silently passed
  off as the source. It's a provenance-honesty rule about the displayed 
  liturgical corpus.

  What it does not forbid: creating commentary. Commentary is new authored
  content, not a stub pretending to be corpus. Nothing in the regime stops us
  from generating it — I conflated "unsourced" with "fake," and those are
  different things.

  The actual consideration with commentary is a different one — authority and
  attribution, which matters specifically because you said in-parish:

  - A commentary pane sitting next to the Vulgate reads as a reference work. In
  a parish study setting it carries teaching weight. AI-generated exegesis there
  is doctrinally unvetted — not fake, but unreviewed — so it should be labeled
  for what it is, and arguably reviewed before a parish leans on it.
  - Whereas the chat mode is already the right home for generated interpretation
  — conversational output is self-evidently the model talking, not the app
  asserting a reference text. Nobody mistakes a chat reply for the Haydock.

  So the real design space is richer than I presented:

  1. Vendored historical commentary (Haydock etc.) — provenance-locked,
  imprimatur-era, renders as a reference layer. Same regime as the Bibles.
  2. Generated commentary/study materials — entirely permitted: build-time
  generation of study guides, discussion questions, session programming per
  passage — shipped after you've eyeballed them, attributed as generated. Or
  generated on-device/on-demand, labeled as such.
  3. Both: Haydock as the canonical reference layer, generated material as the
  parish programming layer (session plans, discussion questions, handouts) —
  which honestly is what "suggested programming and accompanying materials"
  sounds like anyway; Haydock doesn't contain lesson plans.

  My revised recommendation is option 3, with generated parish materials
  produced at build/curation time (so you review once, they ship as static
  curated content with a "prepared with AI assistance" attribution) rather than
  hallucinated live per session — and live generation staying in the chat where
  its nature is obvious.
